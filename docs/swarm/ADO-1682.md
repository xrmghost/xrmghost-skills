# ADO-1682 — xrmghost-skills public discovery surface

## Goal

Improve `xrmghost-skills` as a public discovery surface by updating repo-local public markdown and
metadata guidance with differentiated first-party backlinks, without changing any skill behavior or
runtime semantics.

## Scope and changes

Editorial-only changes to five already-tracked public markdown files. No file under `skills/**`,
`docs/swarm/**` (other than this new doc), or `.github/workflows/**` was touched. No runtime,
build, or executable asset was modified.

- `README.md`
  - Added a new paragraph right after the opening line pointing readers who want product context
    (not source) to the docs site's dedicated Skills section: `docs.xrmghost.tech/skills/`,
    `.../skills/installing-skills/`, `.../skills/using-skills/`. This is a more specific,
    differentiated destination than the generic root/getting-started links already present.
  - In the "Documentation" section: linked the `xg` command reference
    (`docs.xrmghost.tech/cli/reference/`), added a cross-link from `docs/adr/` to the product-wide
    `docs.xrmghost.tech/architecture/` page (explaining the ADRs are repo-local rationale that must
    stay consistent with product architecture), and added a link to `CONTRIBUTING.md` for readers
    who want to change or add a skill.

- `CONTRIBUTING.md`
  - Added an intro paragraph (before "Prerequisites") explaining that skill content encodes the
    product's local dev loop documented at `docs.xrmghost.tech/skills/using-skills/` and the
    `docs.xrmghost.tech/cli/reference/` command surface — with the point that drifting from either
    is a behavior bug regardless of linter status. Also routes product bugs/feature requests to
    `docs.xrmghost.tech/community/report-a-bug/` and `.../request-a-feature/` instead of a PR here,
    to keep this repo's contribution channel scoped to skill wording.

- `SECURITY.md`
  - Added a scope-clarifying paragraph at the top: this policy covers only the `xrmghost-skills`
    repo/CI, not `xg`, AKMS, or the Dataverse plugins the skills drive. Linked
    `docs.xrmghost.tech/editions/` as the authoritative source for the license-tier terms
    (`mock`/`auto`/`live`, Pro license) already referenced further down in the file, instead of
    leaving them undefined here.

- `docs/adr/0001-skill-linter.md` and `docs/adr/0002-agent-judge.md`
  - Each ADR got a short, distinct backlink to `docs.xrmghost.tech/architecture/`: ADR 0001 as a
    contextual note under the title (this ADR is only about structural validation, not the
    product-wide architectural contract), ADR 0002 as an additional reference-list entry (the
    consultive/no-gate recommendation must stay consistent with product architecture). Wording
    differs per ADR and is not a duplicate of the README or SECURITY.md copy.

No GitHub-settings-only discovery gaps were identified as blocking for this task; the repo already
has topics/description/homepage-style signals covered by the README content itself. If a maintainer
later wants to also set the GitHub repository "Website" field to `https://docs.xrmghost.tech` and
add topics such as `agent-skills`, `dataverse`, `dynamics-365`, `power-platform`, that is a GitHub
Settings change outside tracked files and is called out here as a follow-up, not implemented in
this change set.

## Acceptance criteria verified

- README and the other directly relevant public entry points (CONTRIBUTING, SECURITY, both ADRs)
  now include differentiated first-party backlinks appropriate for the skills-repo audience — each
  link targets a distinct docs.xrmghost.tech section (Skills, CLI reference, Architecture, Editions,
  Community) chosen for that specific file's purpose, not a single repeated generic link.
- The added copy is written specifically for each file's audience (contributor, security reporter,
  ADR reader) and is not a verbatim copy of the existing README wording or of docs.xrmghost.tech
  copy — confirmed by manual comparison during editing.
- Repo-local metadata hygiene: cross-links between README, CONTRIBUTING.md, and docs/adr/ were
  added as suggested by the existing KB README audit note; the one gap outside tracked files
  (GitHub repo Website/topics settings) is called out above for maintainers, not implemented here.
- No skill prompt, skill trigger, reference file, or workflow file was touched; `git status
  --porcelain` confirms only the five markdown files listed above changed.

## Residual risks

- The GitHub repository settings (homepage/topics/social preview) follow-up noted above remains
  unimplemented by design (outside tracked files) and needs a maintainer with repo admin access.
- If `docs.xrmghost.tech` restructures the `/skills/`, `/architecture/`, `/cli/reference/`,
  `/editions/`, or `/community/*` paths in the future, these new backlinks will need a follow-up
  pass; they were live and returning `200` at the time of this change (2026-07-07).
