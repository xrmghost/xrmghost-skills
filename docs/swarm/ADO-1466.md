# ADO-1466

## Goal

Implementare il workflow `.github/workflows/skill-lint.yml`, aggiornare `CONTRIBUTING.md` e verificare che `skill-lint` sia presente come required status check su `main`.

## Scope e modifiche

- Letto il bundle `ADO-1466-task-context-cycle-2.json` e l'ADR prerequisito `docs/adr/0001-skill-linter.md` dal worktree di ADO-1465.
- Creato `.github/workflows/skill-lint.yml` con trigger solo su `pull_request` verso `main`.
- Integrato `agent-ecosystem/skill-validator` in modalita` `validate structure -o json skills`.
- Aggiunto conversion step inline da JSON a SARIF 2.1.0 con annotazioni GitHub Actions e upload `github/codeql-action/upload-sarif@v3` con categoria `skill-spec`.
- Aggiunto step finale che forza `exit 1` se il SARIF contiene finding di livello `error`.
- Creato `CONTRIBUTING.md` con istruzioni per bootstrap locale del linter, lettura del JSON/SARIF e consultazione dei risultati nel Security tab.

## Risultati dei test

### Baseline valida

- `skill-validator` sul path `skills/` → exit 0
- output JSON multi-skill generato senza errori o warning sul baseline corrente
- conversione JSON→SARIF locale → `runs[0].results = []`

### Fixture invalida

- fixture temporanea derivata da `skills/debugging-dataverse-plugins-locally/SKILL.md`
- rimosso `name` e svuotato `description` nel frontmatter
- `skill-validator` sulla fixture → exit 1
- conversione JSON→SARIF locale → risultati con finding `Frontmatter` per `name is required` e `description is required`
- il contatore finale degli errori SARIF risulta > 0, coerente con il comportamento atteso del gate bloccante

### Branch protection

- `gh api repos/xrmghost/xrmghost-skills/branches/main/protection` conferma `required_status_checks.contexts = ["scan", "skill-lint"]`

## Summary human-readable

Il workflow `skill-lint` ora valida solo la struttura delle skill su `pull_request` verso `main`, usando il tool scelto in ADO-1465 (`agent-ecosystem/skill-validator`) nel modo piu` conservativo rispetto al requisito zero false-positive. Poiche' il tool non produce SARIF nativamente, il workflow converte il JSON del validator in SARIF 2.1.0 dentro la pipeline stessa, pubblica il risultato nel Security tab e genera anche annotazioni GitHub Actions sul file/linea interessati. Il gate finale fallisce esplicitamente solo in presenza di finding di livello `error`.
